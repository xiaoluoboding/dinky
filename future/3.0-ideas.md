# Dinky 3.0 — Ideas

What earned 2.0: a second and third *media type* (video + PDF joined images). It was a scope change, not just a feature addition. 2.10 already stretched that logic — audio is arguably a fourth media type.

A 3.0 bump should feel *structurally* different to the user: a new surface, a new interaction paradigm, or a new deployment model. More codecs, formats, or settings controls are 2.x point releases.

---

## Candidate directions

### 1. System-wide compression layer

Compression happens automatically on any file regardless of origin — Shortcuts, Share Sheet, Quick Action, clipboard, API — without needing to open the Dinky window. Dinky becomes ambient infrastructure on the Mac rather than a drop zone you visit.

What this requires:
- Proper Share Extension (`.xpc` service, not just a Quick Action)
- Action Extension for Files app
- Shortcuts `AppIntent`s that return output files, not just trigger
- A persistent background agent (SMAppService) that processes a queue even when the main window is closed

### 2. Native Finder integration (status badges + inline compress)

Not a Quick Action — actually native to the file browser. Compression metadata (saved %, output format) shows as a Finder comment or badge on already-compressed files. Right-click context menu offers **Compress with Dinky** in place without opening the app.

What this requires:
- FinderSync extension
- Metadata stored in extended attributes (`com.dinkyfiles.compressionSavings`, etc.)
- Re-compress / revert detection (compare inode + mod date to stored result)

### 3. Local LAN relay (machine-to-machine, no cloud)

Drop files on machine A, compressed output appears on machine B (or a shared folder on a NAS). Strictly local network — Bonjour discovery, no accounts, no upload. Makes Dinky useful in design-to-developer or photographer-to-retoucher handoff workflows.

What this requires:
- `Network.framework` peer-to-peer or Bonjour + TCP
- A lightweight transfer protocol (probably just HTTP over loopback extended to LAN)
- Compression result routing: "send output to…" preset destination that is a remote peer
- Security: codesigned peers only, or PIN pairing

### 4. First-class daemon / headless mode

Right now `dinky serve` is a footnote for pro users. Make it a real first-class feature: Dinky as a macOS daemon that runs compression jobs on schedule, on folder events, or on an HTTP trigger — with a proper queue, retry logic, and a JSON log. Closes the gap with self-hosted automation tools like Hazel.

What this requires:
- LaunchAgent plist for `dinky serve` as an always-on daemon
- Queue persistence (SQLite or flat JSON) so jobs survive restarts
- Health endpoint (`GET /v1/health`) and structured logs
- Optional menu-bar status indicator without the main window

---

## What probably doesn't qualify for 3.0

- More audio codecs or formats
- Video trimming or clip extraction
- PDF merging or splitting
- Better presets UI / import-export presets
- Additional image formats (JPEG XL, etc.)
- More FPS cap options
- Performance improvements

These are all good 2.x releases. 3.0 needs a new *surface* the user interacts with, not a wider palette of the existing surfaces.

---

## Notes on bundle size

Any 3.0 direction must still respect the CLAUDE.md constraint. The most bundle-safe directions are 1 (Share Extension — OS-provided UI, no new dependencies) and 4 (daemon is just the existing binary in a different launch mode). LAN relay (#3) needs careful scoping — Network.framework is free, but any serialisation library is not.
